Eh2406 (Mar 28 2022 at 21:22):

Without/Before specialization...
Fundamentally to_string just calls Formatter which can potentially do a lot of different things. It processes an entire DSL as its first argument and then can pad strings and format numbers in several different notations and ... Furthermore it's some of the oldest code still in STD, and not the most optimized.
Whereas to_ownedwhen it's done doing its desugaring is a specialized function for allocating a new string with the same contents as the old string. It's just an allocation and a memcopy.

Eh2406 (Mar 28 2022 at 21:29):

"specialization" is the feature that lets you have more than one implementation depending on what type it is.
One for a type T where the impl has default fn
and a specialized one for type str for when T happens to equal str.
This functionality works correctly and reliably. But it has not been stabilized (I.E. can only be used by STD) because it is possible to write safe code that leads to undefined behavior.

Eh2406 (Mar 28 2022 at 22:50):

If you look at STD, it only has overrides for char u8 i8 str and String. The specialization optimization only hits if you can make T one of those types. From has nothing to do with it.
@Connor Horman is pointing out that if he did not have Display for StringView then StringView would get defered into an str, and hit the happy path. But that would have other negative implications, specifically that a StringView cannot be displayed, only a &*StringView. Which is probably going to be really annoying when generics are involved.
