Topic: on working groups and teams

nikomatsakis (Jun 23 2020 at 21:54):

I've been thinking about it. It seems to me that there is perpetual confusion about the idea of working groups vs project groups etc. I'm not sure how truly harmful this is but I was thinking about an alternative. Given that we have a notion of "subteams" already (e.g., the dev-tools team has various subteams, etc), maybe we should simplify to:

  • project groups -- execute a defined, often cross-cutting, project with relatively well-defined scope (e.g., project-ffi-unwind)
  • teams and subteams -- indefinite entities that exist to tend to a given area, codebase, etc

nikomatsakis (Jun 23 2020 at 21:54):

By this notion, I think we would just make the embedded working group a Rust team.

nikomatsakis (Jun 23 2020 at 21:54):

We could even have a "domain team" that contains subteams that focus on particular areas, but let's not overgeneralize

nikomatsakis (Jun 23 2020 at 21:55):
