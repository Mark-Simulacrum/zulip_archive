mw (Mar 04 2019 at 16:58):

AGENDA

Current Status

  • Profiler records individual events that can be analyzed later (instead of
    just maintaining aggregate values).

  • Analysis moved out of compiler because it's too expensive

  • Many event types that (will) also allow tracking incremental compilation
  • Serializing events as JSON is too slow (which is not unexpected)
  • TODO: Anything else?

Where to track work?

  • GH issues/PRs for individual sub-tasks seem fine to mw

  • WG meeting notes should contain information of what the current tasks are,
    but it's hard to find.

  • TODO: Should we create one of the following?:
    * A GH tracking issue for the entire project
    * A GH tracking issue for just the next milestone (i.e. an MVP)
    * Some kind of status page in the working group directory?
    * A status section of the WG readme?

What does a minimum viable product look like?

  • Some thoughts by mw:
    * Show table with per-query data (percentage of time spent, cache hits, incr. hits)
    * Either
    * provide commandline tool that does compiling + profile postprocessing, or
    * make this view visible on perf.rlo
    * The data format for this view should be stable so the perf.rlo can support
    it without further work causing churn.

  • TODO: Would this be a good MVP? Other proposals?

Next work items

  • Goal: Establish stable separation between recording and postprocessing.
    * Can be achieved by providing stable data format, or
    * by providing a crate that offers a stable interface to the data (something like an iterator / SAX parser)
    * TODO: Find out what a stable format/interface could look like.

  • Goal: Reduce overhead of recording to a minimum
    * TODO: Settle on strategy (see https://github.com/rust-lang/rust/issues/58372)
    * Are there alternatives or concerns?

  • Goal: Come up with a practical postprocessing pipeline / driver program
    * TODO: Start to collect ideas on what this could look like, how to make it comfortable to use.
