Topic: Tuple elements or tuple fields?

isHavvy (Oct 06 2020 at 18:42):

A tuple type is a heterogeneous product of other types, called the elements of the tuple. It has no nominal name and is instead structurally typed.
-- https://doc.rust-lang.org/nightly/reference/types/tuple.html

Anybody have any opinions on not changing that "elements" to "fields"? I don't think I've actually ever seen anybody call them tuple elements outside of this page. (I'm also adding a bunch of obvious but unstated information on tuples to the type and expression pages)

Eric Huss (Oct 06 2020 at 21:06):

"Elements" seems to be a common term for the members of a tuple in other programming language specs and math texts, though there is a bit of fluidity between the terms. I've always felt like "field" is used more in the context of referring to an element (like .0).

isHavvy (Oct 06 2020 at 22:04):

That is field access to me, so of course what it accesses should be called a field, no?

Eric Huss (Oct 06 2020 at 22:14):

:shrug: I've seen it go both ways. For example, C# sort dances between the two (https://docs.microsoft.com/en-us/dotnet/csharp/language-reference/builtin-types/value-tuples). Julia also uses both terms (https://docs.julialang.org/en/v1/manual/types/#Tuple-Types). Erlang uses both (http://erlang.org/doc/programming_examples/records.html). Pierce calls them "fields". Swift uses "elements (https://docs.swift.org/swift-book/ReferenceManual/Types.html). C++ uses "elements". Python typically uses "elements". I don't feel like the terms are completely synonymous, but I struggle to define how they differ.
