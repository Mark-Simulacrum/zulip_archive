Topic: pinned `{get,make}_mut` for `Arc`?

Waffle Lapkin (Jan 03 2022 at 12:22):

It appears as like currently Pin<Arc<T>> is not that useful, since it's not possible to get Pin<&mut T> in safe code. Is it desirable to add pinned versions of get_mut and make_mut to arc?

impl<T> Arc<T> {
    pub fn get_mut_pin<T>(this: &mut Pin<Arc<T>>) -> Option<Pin<&mut T>> {
        // Safety: Pin is `repr(transparent)` and we don't move the value behind arc
        let unpin: &mut Arc<T> = unsafe { mem::transmute(this) };

        let reference = Arc::get_mut(unpin)?;

        // Safety: Arc is pinned
        Some(unsafe { Pin::new_unchecked(reference) })
    }

    pub fn make_mut_pin<T>(this: &mut Pin<Arc<T>>) -> Pin<&mut T>
    where
        T: Clone,
    {
        // Safety: Pin is `repr(transparent)` and we don't move the value behind arc
        let unpin: &mut Arc<T> = unsafe { mem::transmute(this) };

        let reference = Arc::make_mut(unpin);

        // Safety: Arc is pinned
        unsafe { Pin::new_unchecked(reference) }
    }
}

Daniel Henry-Mantilla (Jan 03 2022 at 22:15):

In general, !Unpin + Clone is kind of an antipattern: if something is self-referential, then it shouldn't be Cloneable, unless the newly created instance had a way to feature a "dangling" sentinel to put instead of the self-reference:

use ::core::marker::PhantomPinned as PinSensitive;

struct SelfRef {
    buf: [u8; 1_024],
    thing: ptr::NonNull<[u8]>,
    _may_be_selfref: PinSensitive,
}

// Use a non-ZST static to be able to use ptr-eq to check for dangliness.
static DANGLING_SENTINEL: [u8; 10] = *b"<dangling>";

impl Clone for SelfRef {
    fn clone (self: &'_ SelfRef)
      -> SelfRef
    {
        Self {
            buf: self.buf.clone(),
            thing: DANGLING_SENTINEL.as_slice().into(),
            _may_be_self_ref: <_>::default(),
        }
    }
}

So, with that in mind; are make_mut_pin -like APIs that necessary?

Oh, I didn't know this. Right, the Deref impl for Pin doesn't require P::Target: Unpin. That's a little weird with respect to interiour mutability, but good to know. The consequence of this is that types that allow going from &W<T> to &mut T (mutex, rwlock, etc) can never simultaneously allow structural pinning in the form of Pin<&W<T>> -> Pin<&mut T>.

So the only solution to keeping one type and allowing structural pinning is moving "Is expected to be pinned?" into a type parameter. Like

impl<T> Mutex<T, Unpinned> {
    pub fn lock_pin(self: &Mutex<T, Unpinned>) -> LockResult<MutexGuard<'_, T>> { /* ... */ }
}

impl<T> Mutex<T, Pinned> {
    pub fn lock_pin(self: Pin<&Mutex<T, Pinned>>) -> LockResult<Pin<MutexGuard<'_, T>>> { /* ... */ }
}

Which to me seems weird, considering that usually using Pin doesn't require having an additional type param...
