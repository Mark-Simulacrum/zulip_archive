Topic: round_up and div_round_up

Josh Triplett (Apr 17 2020 at 20:54):

I would be interested in adding methods for integer types, which round up to the next multiple of a given number, or which divide by a given number rounding up to the next multiple. For example:

These would look like this:

        assert_eq!(0u64.div_round_up(4096), 0);
        assert_eq!(1u64.div_round_up(4096), 1);
        assert_eq!(4095u64.div_round_up(4096), 1);
        assert_eq!(4096u64.div_round_up(4096), 1);
        assert_eq!(4097u64.div_round_up(4096), 2);
        assert_eq!(8191u64.div_round_up(4096), 2);
        assert_eq!(8192u64.div_round_up(4096), 2);

        assert_eq!(0u64.round_up(4096), 0);
        assert_eq!(1u64.round_up(4096), 4096);
        assert_eq!(4095u64.round_up(4096), 4096);
        assert_eq!(4096u64.round_up(4096), 4096);
        assert_eq!(4097u64.round_up(4096), 8192);
        assert_eq!(8191u64.round_up(4096), 8192);
        assert_eq!(8192u64.round_up(4096), 8192);

Does that seem reasonable? I see these operations open-coded quite often, and often incorrectly (not accounting for 0 or for overflow).

I've personally been using this trait in my codebase:

pub trait BitOps: Sized {
    const BITS: u32;
    fn sign_extend(self, width: u32) -> Self;
    fn zero_extend(self, width: u32) -> Self;
    fn fits_in_bits_signed(self, width: u32) -> bool;
    fn fits_in_bits_unsigned(self, width: u32) -> bool;
    fn is_aligned(self, align: Self) -> bool;
    fn align_up(self, align: Self) -> Self;
    fn wrapping_align_up(self, align: Self) -> Self;
    fn checked_align_up(self, align: Self) -> Option<Self>;
    fn align_down(self, align: Self) -> Self;
    fn align_offset(self, align: Self) -> Self;
}

Amanieu (Apr 18 2020 at 01:11):

@Josh Triplett Note that on ARM, the page size is not a compile-time constant (though on most systems it is 4K). The proper way of getting the page size on Linux is to read getauxval(AT_PAGESZ). However now the compiler can't know that it is a power of two and will have to generate a division instruction instead of a faster bitwise masking.

Amanieu (Apr 18 2020 at 01:14):

You can, but the problem is that the compiler doesn't know that. So it can't optimize the divide to a mask.
