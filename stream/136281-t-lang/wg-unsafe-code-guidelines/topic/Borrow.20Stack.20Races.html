Connor Horman (May 11 2021 at 14:49):

If we reason about the reborrow as a side-effect on some unspecified memory location &L, then the potential ordering becomes apparent. On thread1, we have the following:

  • atomic side-effect (or rmw) on &L

  • atomic side-effect on M
    On thread2, the sequence-order is:

  • atomic value computation on M

  • (potentially) atomic side-effect (or rmw) on &L.
    In this, it's possible for the side-effect on M from thread1 to become visible to thread2 before the value computation (thus the value computation reads the value the side effect stores), while not making the side-effect on &L visible, allowing the reborrow in thread2 to appear earlier in the modification order of &L, then the reborrow on thread1. I'm not sure of a way to reason about it, other than treating the borrow stack as an invisible memory location that is atomically modified with relaxed ordering.
