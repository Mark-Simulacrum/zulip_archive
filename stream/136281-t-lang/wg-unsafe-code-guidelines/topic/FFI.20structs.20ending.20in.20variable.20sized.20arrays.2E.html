Topic: FFI structs ending in variable sized arrays.

Peter Rabbit (May 11 2020 at 18:28):

Suppose I have a struct defined like so:

#[repr(C)] struct Foo {
    length: u32,
    data: [u8; 1],
}

I now allocate some memory and pass it to a function which fills it in or maybe it allocates some memory for me and returns a raw pointer that I return to another function later to free.
Either way I now have a foo: *mut Foo where the length of foo.data is determined by foo.length. However I feel like doing slice::from_raw_parts(foo.data.as_ptr(), foo.length as usize) is not sound as I'm using a pointer to a smaller data to access a larger chunk of data. Is this sound, and if not what is the correct way to get a slice to the full data?

RalfJ (Jun 11 2020 at 08:07):

So what should definitely work is to avoid the &mut self:

    unsafe fn as_slice_mut<U>(foo_ptr: *mut Foo, base_ptr: *mut U, length: usize) -> &mut [T] {
        let offset = (foo_ptr as isize) - (base_ptr as isize);
        let array_ptr = (base_ptr as *mut u8).offset(offset) as *mut T;
        slice::from_raw_parts_mut(array_ptr, length)
    }

RalfJ (Jun 11 2020 at 08:08):

now, if we had raw_ref, you could get that foo_ptr without creating an intermediate mutable reference and all would be good, but that's still unstable unfortunately... however, since foo_ptr is not used after slice::from_raw_parts_mut, I think calling this like this is actually okay:

fn main() {
    unsafe {
        let mut raw_memory: [u32; 8] = [6, 5, 4, 3, 2, 1, 0, 99];
        let foo: *mut Foo = raw_memory.as_mut_ptr() as *mut Foo;
        let slice_mut = Foo::as_slice_mut(&mut (*foo).data, foo, (*foo).length as usize);
        slice_mut[5] = 200;
        let slice = (*foo).data.as_slice(foo, (*foo).length as usize);
        println!("{:?}", slice);
    }
}

That's definitely a wrong Rust representation of the type, the C type is affirmatively unsized and does not contain a pointer in that position. You need something like this:
