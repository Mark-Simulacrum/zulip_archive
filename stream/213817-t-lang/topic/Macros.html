Would like to share these wonderful examples due to @Daniel Henry-Mantilla and @Kestrer :

mod blob { pub fn starboard() -> &'static str { "blob" }}
dbg!(starboard());
{
    dbg!(starboard());
    use blob::*;
}
let starboard = || "top-level let";
dbg!(starboard());
fn starboard() -> &'static str { "top-level fn" }
dbg!(starboard());
{
    dbg!(starboard());
    fn starboard() -> &'static str { "nested fn after let" }
    {
        dbg!(starboard());
        use blob::*;
    }
}

printing:

[src/main.rs:5] starboard() = "blob"
[src/main.rs:9] starboard() = "top-level let"
[src/main.rs:11] starboard() = "top-level let"
[src/main.rs:13] starboard() = "nested fn after let"
[src/main.rs:16] starboard() = "blob"

Vadim Petrochenkov (Mar 22 2022 at 22:14):

The m!() example looks like a bug, m!() is supposed to behave identically regardless of whether m is resolved directly (the b case) or through an import (the a case).

Vadim Petrochenkov (Mar 22 2022 at 22:21):

The relevant code in rustc is fn resolve_ident_in_lexical_scope, the "resolution at def site" logic for macro_rules triggers only if the macro_rules was introduced with RibKind::MacroDefinition.
This is a legacy implementation that is different from how "resolution at def site" works for macros 2.0, for example.
