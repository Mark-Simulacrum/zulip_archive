Manuel Drehwald (Nov 23 2021 at 20:58):

Hi Nagisa, Josh (and everyone else involved).

After half a year of development, I would like to revisit this decision.
Generally, we have two options. The plugin above, together with the calling convention here: https://enzyme.mit.edu/getting_started/CallingConvention/, or using Enzyme as a library on some LLVM-IR.
The C-API is generally preferable, as it allows more flexibility to add a nice Rust frontend and generate gradients in the most efficient way.
However, here are some disadvantages we encountered (some also apply to the plugin approach):

1) Cargo intentionally doesn't support post-build.rs, however I can only invoke Enzyme after running cargo. Wrapping cargo to run twice or building dependencies which use enzyme in an external dir isn't desirable.
2) We use the C-abi / ffi, in order to create an archive with the gradient functions which we link then. However, Enzyme could support more types (slices, non-repr(c) structs, ..) which would increase the usefulness and performance of the generated code.
Not requiring [no-mangle] and extern blocks would also clean up the Rust api of Enzyme.
3) We need a custom build rustc/llvm/enzyme/clang stack atm., which can't make use of rust's PGO and is an extra layer of complexity.
That's mainly needed since rustc isn't exposing some of the required llvm symbols.
4) We currently rely on dwarf output to analyze the memory layout of types. That's not perfect and especially due to the C-Abi we
already had mismatches, e.g. returning { f32, f32 } might show up as the return of a double value or { T } was simplified to T.
5) I guess there is no real path for stabilization with the current setup (including the -llvm-plugins flag)?

We think that most of these issues should be solvable by moving from a library solution into the compiler, mostly cg_llvm.

  • A single compilation run would be sufficient
  • We could get the type information directly, without using proxies like dwarf
  • We can skip the entire ffi overhead, since we can directly add the generated functions, rather than linking them back.
  • We should be able to access our missing llvm symbols, so can work with the default rustc.

Are there any general issues which you can see with this approach, from the rustc side? Of course we would need to have Enzyme as a part of the llvm monorepo before being able to stabilize something like this (together with a new rfc), but do you have any other, early thoughts?
