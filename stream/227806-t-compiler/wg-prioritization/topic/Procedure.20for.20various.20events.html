Léo Lanteri Thauvin (Jun 11 2020 at 17:31):

According to the Master Document, we should do this:

  • Create a topic per stable/beta nominations, i-nominated and prs-waiting-on-team
    • create a topic once an issue is nominated/PR waiting on team.
    • send a text with directives on what to do on every nomination or PR waiting on team that is like ...
      • Sanity check if the nomination/pr waiting is really for T-compiler/libs-impl
      • We want to add them to the agenda with some basic information about why the issue was nominated/waiting, what's the status, etc
  • Create a topic per P-critical and P-high regressions
    • create a topic once an issue is tagged with P-critical/P-high regression.
    • send a text with directives on what to do on every one of them like ...
      • Ping appropriate people and/or notification groups.
      • Assign if possible; if it remains unassigned, add it to agenda so we can assign during the meeting.
      • Check if there are relevant issues that are worth raising awareness.

So I think we should define the procedure for these events and add that into #73247.

Léo Lanteri Thauvin (Jun 12 2020 at 16:38):

I-prioritize procedure

  • Check if the issue is relevant to the compiler team (compiler or libs-impl issue), if not remove the request for prioritization.
  • If the issue needs bisection and/or MCVE, label it accordingly with E-needs-bisection and E-needs-mcve and ping the Cleanup Crew with @rustbot ping cleanup
  • If the issue is relevant to a specific notification group, e.g. LLVM or Windows related bugs, ping the notification group with @rustbot ping <group>
  • Discuss the priority of the bug with the rest of the WG
