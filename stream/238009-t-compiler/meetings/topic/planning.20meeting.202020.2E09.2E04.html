ok, I spent a lot of today on this.

  • https://github.com/rust-lang/rust/pull/76418
  • https://github.com/rust-lang/rust/pull/76417
  • https://github.com/rust-lang/rustc-dev-guide/pull/852
  • https://github.com/rust-lang/rustc-dev-guide/pull/851
  • https://github.com/rust-lang/rustc-dev-guide/issues/850
  • https://github.com/rust-lang/rust/issues/76419

The tl;dr is that we do a lot of things wrong, and there are a lot of reasons they avoid PRs: https://hackmd.io/RHyLd-27QXOR5AHa-oa6Fg?view. This is not super favorable to the rust project, so I'm hitting the highlights here in a less frustrated tone. Apologies for the wall of text.

  • Compile times are really bad, especially on first builds. https://rust-lang.zulipchat.com/#narrow/stream/131828-t-compiler/topic/improving.20bootstrap.20times.20for.20contributor.20fun.20and.20profit already discussed this a lot. The space taken for a compile is also really bad.
  • Taking extensive knowledge of git as a prerequisite makes it a lot harder to contribute. I opened https://github.com/rust-lang/rustc-dev-guide/issues/850 for this, @WG-rustc-dev-guide can fix this ourselves without too much trouble I think.
  • The x.py defaults are meant for CI, not for contributors. https://github.com/rust-lang/compiler-team/issues/326 helped with this, but we should try not to have to recommend changes to config.toml.example in the dev guide.
  • Review latency is bad. https://rust-lang.zulipchat.com/#narrow/stream/131828-t-compiler/topic/.22Review.20latency.20can.20be.20challenging.22 talked about this, but I don't know that there were any changes as a result of the discussion.
  • Merge conflicts are frustrating. This compounds with the git issues and review latency; it also doesn't help that the compiler changes so frequently.
  • The dev-guide has room for improvements:
    • It could be organized better (will write this up in an issue at some point)
    • It doesn't have enough information. In particular, a lot of knowledge seems to only live in the heads of contributors (e.g. https://github.com/rust-lang/rustc-dev-guide/issues/836). We need more compiler members working on this, not just people from the working group.
  • Being able to build libstd with just cargo build and cargo test would make it a lot easier to get started.
  • It's hard to find mentors. This compounds with 'knowledge only lives in the heads of contributors'.

Some other pieces of knowledge:

  • New contributors definitely use CI rather than having to build locally
  • First-time build times are a major issue, not just incremental builds
  • S-waiting-for-help would help new contributors the most. Maybe we should reconsider this?
  • New contributors don't know rustbot exists and they can modify labels. I think this could be fixed by mentioning it in the highfive welcome message.
