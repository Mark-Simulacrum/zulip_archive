Kirill Bulatov (Oct 06 2021 at 15:00):

A few ideas/wishes that not necessarily medium-sized, but might be fun:

  • improve closure syntax completions:
    * now, when you write something like .map(|a| <|>), you have to mess a lot with | — they are not added automatically, not closed automatically, when there are some complex params instead of a (tuples, for instance), it becomes a pain to destucture this all without confusing the brackets
    * .map(function<|>) will now result in .map(function()) always while there are plenty of cases when it could be .map(function) or .map(|a| function(a)), but completed automatically for you

  • support "live" files addition to the workspace: currently, we are allowing to _start_ VSCode with a single arbitrary rust file outside any workspace and it will get some IDE features. But there's no way to support more than one or just add a standalone file into an open workspace to get some IDE features in it. Here's why: https://github.com/rust-analyzer/rust-analyzer/pull/8955#issuecomment-916119471

  • move the inlay hints LSP support further: there's an unfortunately stalled PR https://github.com/microsoft/language-server-protocol/pull/1249 (most probably more than one) and great confusion for many people on "why does TypeScript have it in VSCode but Rust does not?" topic. And a great push towards the feature adoption in our codebase.

  • support "compose Cargo task to run for a workspace" task: given a certain workspace, add an action that allows to input characters, which are fuzzily matched against all possible cargo command outputs with options, something like this:
    image.pngijr.png
    The point is to create a run configuration out of this to allow the VSCode to rerun the command with a single shortcut which is super useful for cases like "quickly compose a cargo clippy command to test this module and run it N times".

Kirill Bulatov (Oct 06 2021 at 15:06):

Another one that might be small, but somewhat tricky to find the actual issue:

We now have a decent mod <|> support: we actually check if there's anything on the FS that's not yet added and suggest it.
But

  • it all breaks with any letter added afterwards mod a<|> will not give you anything, even if there are suitable files on the disk
  • when you type mod <|>, no completion is actually proposed, you have to force it with cmd/ctrl + space which hurts the feature's discoverability abysmally :smile:
