Topic: Process for unmaintained crates

Danilo (Jan 24 2022 at 16:31):

I just saw the RustSec "unmaintained" advisory for stdweb. Too bad...

I don't know if there are certain formal rules already, but I wonder if there should be a process for dealing with unmaintained crates. For example by marking stdweb as unmaintained, a lot of people will probably try to move away from it. However, after determining that the crate is indeed unmaintained (e.g. by posting a formal "request for clarification of maintenance status" with a waiting time of 1-2 months) maybe there should be a "call for new maintainers" first? For some crates, a new team of maintainers might form, that could fork and pick up maintenance from there.

Of course, if the original maintainer is not reachable, there would still need to be a transition of the old crate to the forked one. However, then the advisory could actually point to the fork, instead of just saying "this crate is unmaintained, sucks to be you".

Danilo (Jan 24 2022 at 16:32):

That process could also include some recommendations for developers of popular crates that are close to their limits, or that need additional help in order to keep maintaining a certain crate. Recommendations could include "publish a 'call for maintainers' issue with the following content and link to it from your README".

Shnatsel (Jan 28 2022 at 23:54):

So the topics I already covered are:

  • Monotonic fields on the "vulnerability" and "advisory" GraphQL entities, so that clients could process new advisories since they last synced, and implement efficient sync
  • Frequency of GHSA's import from RustSec database - last time I checked there was a large delay before new RustSec data shows up in GHSA
  • Support for converting ^ and ~ version selectors that RustSec allows to GHSA (historically there were issues with that)

Didn't get a chance to cover yet:

  • Authoritative data for a given GHSA ID - curated vs original advisory, only original showing up in Google search results
  • CC-BY-SA licensing and exact attribution requirements
  • Better repository integration for end users affected by an advisory (dependabot?)
